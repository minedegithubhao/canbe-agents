# RAG 检索评估页面原型

## 1. 页面定位

RAG 检索评估页面用于管理评估集、生成评估案例、运行检索评估、查看历史评估结果。

第一版只评估检索质量，不评估答案生成质量，不接入 RAGAS，不引入 LLM Judge。

核心评估对象是 chunk，不是 FAQ。

## 2. 主页面结构

主页面采用“评估集列表 Table”作为核心结构。

```text
┌────────────────────────────────────────────────────────────────────┐
│ RAG 检索评估工作台                                                  │
│ 用于验证当前 RAG 检索配置是否能召回正确 chunk                       │
│                                                                    │
│ [一键生成评估集] [刷新]                                             │
└────────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────────────┐
│ 评估集列表                                                          │
├──────────────┬──────────────┬──────────┬────────────┬──────────────┤
│ 评估集名称    │ 数据源        │ 案例数    │ 创建时间     │ 操作          │
├──────────────┼──────────────┼──────────┼────────────┼──────────────┤
│ jd_help_v1   │ cleaned.jsonl │ 100      │ 2026-05-13 │ [开始评估]     │
│              │              │          │            │ [评估记录]     │
│              │              │          │            │ [删除]         │
├──────────────┼──────────────┼──────────┼────────────┼──────────────┤
│ jd_help_v2   │ cleaned.jsonl │ 300      │ 2026-05-13 │ [开始评估]     │
│              │              │          │            │ [评估记录]     │
│              │              │          │            │ [删除]         │
└──────────────┴──────────────┴──────────┴────────────┴──────────────┘
```

设计理由：

- 评估集是主对象，生成、运行、历史、删除都应该围绕评估集展开。
- 评估记录是评估集的从属对象，不直接铺满首页。
- 后续新增复制评估集、导出结果、对比评估记录时，可以自然挂到操作栏或历史弹层中。

## 3. 一键生成评估集弹窗

一键生成评估集采用 3 步式向导。

```text
┌────────────────────────────────────────────────────────────┐
│ 一键生成评估集                                              │
│ 步骤：① 基础信息 → ② 抽样策略 → ③ 生成预览与确认              │
└────────────────────────────────────────────────────────────┘
```

### 3.1 基础信息

```text
评估集名称      [jd_help_eval_v1]
生成数量        [100]
数据源          exports/jd_help_faq.cleaned.jsonl
数据源指纹      sha256: xxxxx

[取消] [下一步]
```

字段说明：

- 评估集名称：用于列表展示和人工识别。
- 生成数量：决定本次评估集包含多少条案例。
- 数据源：当前用于生成案例的 clean 数据来源。
- 数据源指纹：生成时保存 `source_hash`，运行评估前实时重新计算并校验。

### 3.2 抽样策略

#### 3.2.1 评测类型 eval_type

```text
[ ] single_chunk
[ ] multi_chunk
[ ] single_chunk + multi_chunk
```

说明：

- `single_chunk`：一个问题期望命中一个核心 chunk，用于验证基础召回能力。
- `multi_chunk`：一个问题需要命中多个相关 chunk，用于验证复杂问题的多证据召回能力。
- `single_chunk + multi_chunk`：混合生成两类案例。

推荐默认值：

```text
single_chunk + multi_chunk

single_chunk   70%
multi_chunk    30%
合计：100%
```

推荐理由：

- 单 chunk 案例能验证基础检索是否可用。
- 多 chunk 案例能提前暴露复杂问题下的召回不足。
- 70/30 比例适合 MVP：不会让多 chunk 案例过多导致第一版解释成本过高。

#### 3.2.2 问题改写方式 question_style

```text
original      [30%]
colloquial    [40%]
synonym       [20%]
abbreviated   [10%]
合计：100%

[恢复推荐值]
```

说明：

- `original`：使用原始问题或接近原始问题。
- `colloquial`：口语化改写，用于测试用户真实表达。
- `synonym`：同义改写，用于测试语义表达变化。
- `abbreviated`：省略式表达，用于测试信息不完整时的检索鲁棒性。

设计理由：

- `question_style` 用于观察检索系统对不同表达方式的稳定性。
- 允许手动调整，可以让用户构造更接近真实线上流量的评估集。

#### 3.2.3 难度 difficulty

```text
easy      [30%]
medium    [50%]
hard      [20%]
合计：100%

[恢复推荐值]
```

说明：

- `easy`：问题表达清晰，和目标 chunk 高度接近。
- `medium`：问题经过改写，但核心语义明确。
- `hard`：问题表达更间接，可能需要多个 chunk 或更强召回能力。

设计理由：

- `difficulty` 用于观察检索质量是否只在简单问题上表现好。
- 手动调整后，可以针对薄弱场景单独构造压力测试集。

#### 3.2.4 类别分布 category

```text
订单相关      [35%]
退款售后      [25%]
物流配送      [20%]
账户支付      [10%]
发票相关      [10%]
合计：100%

默认值：根据 clean 数据真实分布自动计算
[恢复真实分布]
```

说明：

- MVP 阶段 `category` 类别选项先写死在前端页面。
- 前端写死的类别选项来自当前 RAG 数据中的分类信息。
- `category` 默认比例来自 clean 数据真实分布。
- 用户可以手动调整各类别占比。

设计理由：

- 默认真实分布可以让评估集更接近当前知识库结构。
- 手动调整可以让用户针对重点业务类别增加样本。
- 类别选项先写死在前端，可以降低 MVP 的接口复杂度。
- 后续如果分类经常变化，再改为后端提供 category 字典接口。

### 3.3 生成预览与确认

```text
预计生成：100 条

eval_type：
single_chunk 70 条
multi_chunk 30 条

question_style：
original 30 条
colloquial 40 条
synonym 20 条
abbreviated 10 条

difficulty：
easy 30 条
medium 50 条
hard 20 条

category：
订单相关 35 条
退款售后 25 条
物流配送 20 条
账户支付 10 条
发票相关 10 条

示例 single_chunk 案例：
case_id: faq_eval_000001
question: 下单后还能改规格吗？
eval_type: single_chunk
question_style: colloquial
difficulty: medium
category: 订单相关
expected_retrieved_chunk_ids:
- chunk_订单相关_2_001

示例 multi_chunk 案例：
case_id: faq_eval_000002
question: 已付款但还没发货，想改地址和规格怎么办？
eval_type: multi_chunk
question_style: colloquial
difficulty: hard
category: 订单相关
expected_retrieved_chunk_ids:
- chunk_订单相关_2_001
- chunk_订单相关_5_001

[上一步] [确认生成]
```

设计理由：

- 预览让用户在生成前确认抽样结果是否符合预期。
- 示例案例让用户理解评估集会生成什么结构的数据。
- `expected_retrieved_chunk_ids` 是后续计算 Hit@K、Context Recall@K、MRR@K、Precision@ConfiguredK、Precision@EffectiveK 的核心标准答案。

## 4. 校验规则

生成前必须校验：

- `eval_type` 分布合计为 100%。
- `question_style` 分布合计为 100%。
- `difficulty` 分布合计为 100%。
- `category` 分布合计为 100%。
- 生成数量必须大于 0。

如果校验不通过，禁止进入生成预览或确认生成。

## 5. 操作栏行为

评估集 Table 的操作栏包含：

- 开始评估：对当前评估集启动一次检索评估。
- 评估记录：打开当前评估集的历史评估弹出层。
- 删除：删除当前评估集及其关联案例。是否级联删除历史 run 结果需要在实现计划中明确。

## 6. 开始评估流程

```text
用户点击某个评估集的 [开始评估]

1. 后端读取评估集的 source_path 和 source_hash。
2. 后端重新计算当前 source_path 对应文件的 hash。
3. 如果 hash 不一致，禁止运行，并提示重新生成评估集。
4. 如果 hash 一致，开始运行检索评估。
5. 评估完成后保存 eval_run 和 eval_run_results。
6. 页面展示本次评估摘要，并允许进入评估记录详情。
```

## 7. 评估记录弹出层

点击 Table 操作栏中的“评估记录”后，打开弹出层。

```text
┌────────────────────────────────────────────────────────────────────┐
│ 评估记录：jd_help_eval_v1                                           │
├────────────────────────────────────────────────────────────────────┤
│ Run ID       │ 时间        │ Hit@K │ Recall@K │ MRR@K │ P@CK │ P@EK │ 操作 │
├──────────────┼────────────┼───────┼──────────┼───────┼──────┼──────┼──────┤
│ run_001      │ 05-13 10:20 │ 86%   │ 79%      │ 0.68  │ 31%  │ 52%  │ 详情 │
│ run_002      │ 05-13 11:30 │ 91%   │ 84%      │ 0.73  │ 34%  │ 57%  │ 详情 │
└──────────────┴────────────┴───────┴──────────┴───────┴──────┴──────┴──────┘
```

设计理由：

- 历史评估是从属信息，放在弹出层可以保持主页面简洁。
- 多次评估记录并列展示，后续可以扩展为“对比两次运行”。

## 8. 评估详情弹出层

点击某条评估记录的“详情”后，打开评估详情弹出层。该弹出层用于回答两个问题：

- 本次评估整体效果如何？

```text
┌────────────────────────────────────────────────────────────────────┐
```

## 9. 已确认决策

- 主页面以评估集 Table 为核心。
- “一键生成评估集”采用 3 步式向导。
- `eval_type` 支持 `single_chunk`、`multi_chunk`、`single_chunk + multi_chunk` 三种选择。
- 当选择混合类型时，允许配置 `single_chunk` 与 `multi_chunk` 的比例。
- `question_style` 允许手动调整，默认使用推荐比例。
- `difficulty` 允许手动调整，默认使用推荐比例。
- `category` 类别选项 MVP 阶段先写死在前端页面，类别来源是当前 RAG 数据中的分类信息。
- `category` 允许手动调整，默认比例来自 clean 数据真实分布。
- 所有分布配置必须校验合计为 100%。
- 字段名统一使用 `multi_chunk`，不使用错误拼写 `mutl_chunk`。
- 评估详情弹出层采用“整体指标 + 诊断筛选 + Case 明细”的结构。
- Case 详情采用诊断抽屉展示，包含问题、基础信息、单 case 指标、期望 chunk、实际召回 chunk 和诊断结论。
